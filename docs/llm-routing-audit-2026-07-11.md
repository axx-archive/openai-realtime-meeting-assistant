Produced by a 14-agent gated workflow (4 code auditors, 2 web researchers, 2 designers,
adversarial review gate with 3 repair agents). Review rounds: 8.3 → repairs → 8.2 → repairs
(all flagged gaps closed; final formal re-score aborted by session rate limit — clauses

## Executive summary

1. **Architecture**: Bonfire is ONE Go binary with a fully hand-rolled agentic loop —
   no Claude Code, no Agent SDK, no vendor SDKs at all. Two wire seams: Anthropic
   Messages (streaming SSE, agent_runner_anthropic.go:296) and OpenAI Responses
   (openai_responses.go:65). The Fable orchestrator (anthropicFableRunner) is a bounded
   24-turn tool loop, effort=high, ~20 in-process tools, refusal fallback to Opus 4.8.
2. **Realtime ↔ Fable**: DIRECT, in-process. gpt-realtime-2 has function tools
   (launch_agent_thread / initiate_goal / propose_codex_task); a spoken "kick off X"
   emits a function_call → kanban.go handleToolCall → launchAgentThreadWithSpec →
   selectAgentRunner → anthropicFableRunner (Fable 5). Result returns to the voice as
   function_call_output; the Fable run proceeds outside the voice loop, delivering via
   artifact metadata + notifications. Secondary indirect path: transcripts → brain →
   board worker → propose_codex_task → human confirm → same runner.
3. **GLM 5.2 verdict: DO NOT build the Venice integration now.** GPT-5.6 Luna ($1/$6,
   GA 2026-07-09) is CHEAPER than GLM 5.2 ($1.40/$4.40) on the exact lanes GLM would
   serve, and is a zero-code env flip (OPENAI_BRAIN_MODEL). GLM 5.2 on Venice is beta,
   FP8, Pro-gated, no SLA, no vision, with host-dependent tool-call reliability (~8%
   vs ~1% error rates across hosts per OpenRouter). Keep GLM 4.7 Flash ($0.13/$0.50)
   as a conditional future pilot on self-healing hygiene lanes only, behind the
   telemetry ledger. The venice_chat.go design (below) is shelf-ready if economics flip.
4. **The real money is env vars, not new providers** (in ROI order):
   OPENAI_BRAIN_MODEL=gpt-5.6-luna (~80% off the 11-lane ambient fleet);
   OPENAI_BOARD_MODEL=gpt-5.6-terra; revert prod transcription pin gpt-realtime-whisper →
   gpt-4o-transcribe (65% cheaper AND restores domain-vocab biasing); verify/set
   OPENAI_REALTIME_REASONING_EFFORT=low; consider BONFIRE_REVIEW_MODEL=claude-sonnet-5.
5. **Prerequisite**: the system parses token usage then throws it away (Anthropic input
   tokens dropped; OpenAI usage never parsed). Build usage_ledger.go (per-call JSONL:
   provider/model/seat/tokens/cost) BEFORE any routing change — the agentic-CFO lens
   currently has no books.
6. **Spend model (estimates from code constants)**: ~$4.75 per active meeting-hour
   typical; ~$380/mo light (2 mtg-hr/day) to ~$1,500/mo heavy (8 mtg-hr/day).
   Calendar risk: Sonnet 5 intro pricing expires 2026-08-31 (+50% on chat/router seats).

## AUDIT: runnerAudit

**Summary:** The agent runner layer is a fully hand-rolled, in-process agentic loop — no anthropic-sdk-go, no OpenAI SDK, no Claude Code or Agent SDK integration (go.mod's only deps are WebRTC/pion, webauthn, webpush, crypto). All LLM traffic goes through two bespoke HTTP seams: a streaming SSE client for Anthropic's Messages API (agent_runner_anthropic.go:296, always stream:true, accumulated back into a non-stream response shape) and a plain JSON client for OpenAI's Responses API (openai_responses.go:65, hardcoded api.openai.com URL). The flagship runner is anthropicFableRunner: a bounded multi-turn tool loop (24 turns, 24 server-tool pauses, 15m timeout) on claude-fable-5 at effort=high, dispatching ~20 allowlisted in-process Go tools plus a report_goal_state control tool, with prompt-cache breakpoints, per-tool-result 24K-char caps, refusal fallback to claude-opus-4-8, and pause_turn resume for server web_search/web_fetch (pinned to the basic 20250305/20250910 variants so the raw loop can resend history verbatim). Model routing follows a hardcoded "doctrine": Fable 5 orchestrates, claude-opus-4-8 reviews/gates, claude-sonnet-5 fronts chat/follow-ups, Haiku is refused by doctrineModelOrDefault, and effort below medium clamps up. Provider abstraction (AgentRunner: Name/Capabilities/RunJob→progress channel) is a clean seam with 5 implementations (anthropic_fable, codex_sidecar, codex_local, openai_text, stub), but selection is a closed enum wired through four switch statements — a Venice/GLM OpenAI-compatible provider cannot be registered today without code changes (no runner registry, no base-URL override, and the existing OpenAI path speaks the Responses API, not chat-completions). Orchestration is decided by env dials (BONFIRE_AGENT_RUNNER / BONFIRE_EXECUTION_RUNNER / BONFIRE_*_MODEL) plus the /goal engine, which keyword-matches subtask text to assign runners and stamps per-subtask overrides via artifact metadata.

**Architecture notes:**
- WHAT THIS IS: a fully custom agentic loop over hand-rolled HTTP — NOT Claude Code, NOT the Claude Agent SDK, NOT the Anthropic tool runner. go.mod (module github.com/openai/openai-realtime-meeting-assistant) contains zero LLM SDKs; the Anthropic wire layer is ~380 lines of bespoke code: buildAnthropicMessagesPayload (manual JSON with cache_control stamping), createAnthropicMessagesResponseHTTP (always stream:true, x-api-key + anthropic-version 2023-06-01 headers), and decodeAnthropicSSEStream (a hand-written SSE accumulator that folds message_start/content_block_start/delta/stop events back into the non-streaming response shape, preserving unknown block fields raw so thinking blocks round-trip byte-for-byte).
- THE LOOP (anthropicFableRunner.RunJob, agent_runner_anthropic.go:725-942): build system prompt (Scout persona + ten-step goal loop + authority ladder + mode contract + terminal-output law) and one user turn → per turn: call Messages API; on stop_reason=tool_use, decode content blocks, execute client tools in-process via app.applyToolCallArgs against a curated allowlist (orchestratorToolAllowlist: ~20 board/memory/package/notification/fiscal.ai tools; create_ticket forced to draft=true; note_for_the_record intercepted for author provenance), answer all tool_use blocks in ONE user turn, echo the assistant turn verbatim; on pause_turn (server web tools), echo verbatim with NO user turn and decrement a separate 24-pause budget (doesn't consume the 24-turn budget); on end_turn, terminal; on refusal, retry once on claude-opus-4-8 with the conversation unchanged; on max_tokens/other, honest incomplete (needs_attention/blocked, never verified). Progress streams to the engine as an AgentProgress channel (stage/percent/note/gate via the report_goal_state control tool + monotonic high-water heuristic), not token deltas.
- CONTEXT MANAGEMENT: no compaction, no summarization, no context editing. Instead: (1) prompt caching — ephemeral cache_control breakpoints on the last tool, the system prompt, and the newest two user turns (max-4-breakpoint budget, image blocks never marked, assistant turns never touched); (2) tool-result capping at 24000 chars (voice twin 8000) with an honest elision marker; (3) request-level image caps (12 images / 20MB decoded); (4) the conversation simply grows until maxTurns=24. Server web tools are deliberately pinned to the BASIC variants (web_search_20250305 / web_fetch_20250910) because the _20260209 dynamic-filtering variants embed code_execution blocks that 400 when this raw loop resends history verbatim to resume a pause_turn — a real constraint of not using an SDK.
- SUBAGENTS: none inside the runner. Multi-agent behavior lives one layer up in the /goal engine (goal_engine.go): decompose (Fable call) → assignGoalRunners stamps a runner per subtask → dispatchReady launches child agent threads (concurrency BONFIRE_GOAL_CONCURRENCY=2, per-user goal cap 2) → each child is a full anthropicFableRunner run → review/gate calls on the dedicated review model (claude-opus-4-8) → fold results back into the parent plan. A Fable→Sonnet worker fan-out is documented as a 'sanctioned future seam (designed, NOT built)'.
- PLUGGABILITY — the interface is clean, the registration is closed. AgentRunner (agent_runner_iface.go:77) is 3 methods: Name(), Capabilities() (CanShell/CanBrowse/CanEditRepo/CanCommit/ToolLoop/MaxRuntime), RunJob(ctx, job) → <-chan AgentProgress. Five implementations exist (anthropic_fable in-process loop; codex_sidecar/codex_local wrapping the external Codex CLI; openai_text single-completion; stub). BUT there is no registry: runner names are a const enum (agent_runner_iface.go:85-91) and adding a provider requires editing four hardcoded switches — selectedAgentRunnerName (env alias matrix), selectedExecutionRunnerName, selectAgentRunner (name→constructor), and resolveAssignedRunnerName (per-subtask override validation, which silently drops unknown names).
- COULD VENICE/GLM BE REGISTERED TODAY? No — three gaps: (1) no runner registry or dynamic name acceptance (an assignedRunner value of 'venice' resolves to "" and falls back to the env default); (2) the existing OpenAI path can't be repointed — openAIResponsesURL is a hardcoded const targeting OpenAI's *Responses* API (no OPENAI_BASE_URL/base-url override), while Venice/GLM speak OpenAI *chat-completions*, so a new responder + wire type is needed either way; (3) the model-doctrine guard (doctrineModelOrDefault) only governs Anthropic env dials. WHAT'S ACTUALLY NEEDED (small): a new struct implementing the 3-method interface emitting one terminal AgentProgress (the openAITextAgentRunner wrapper in agent_runner_providers.go is a 27-line template), a name constant, and entries in the four switches. The channel-based progress contract, capability struct, and per-subtask assignedRunner metadata would all work unchanged.
- ORCHESTRATION / WHO PICKS THE MODEL: three layers. (a) Env dials resolved at construction time: BONFIRE_AGENT_RUNNER (orchestrator runner; default anthropic_fable iff ANTHROPIC_API_KEY set, else legacy BONFIRE_AGENT_THREAD_WORKER/BONFIRE_CODEX_AGENT_THREADS worker), BONFIRE_EXECUTION_RUNNER (default codex_sidecar), plus per-seat model dials BONFIRE_ORCHESTRATOR_MODEL / BONFIRE_REVIEW_MODEL / BONFIRE_CHAT_MODEL / BONFIRE_FALLBACK_MODEL / OPENAI_BRAIN_MODEL. (b) The /goal engine: assignGoalRunners keyword-matches subtask title+detail against a phrase list ('run the tests', 'write code', 'git ', ...) to route shell/repo subtasks to the execution runner — NOTE: despite the AgentCapabilities struct existing 'so the /goal engine reads these', assignment today is the keyword heuristic, not a Capabilities() lookup. The assignment is stamped as artifact metadata assignedRunner and honored per-job in selectAgentRunner; deliverable subtasks get heavier budgets via the goalDeliverable metadata flag → job.Effort/job.MaxTokens overrides. (c) Hardcoded routing doctrine (founder directive 2026-07-07, agent_runner_anthropic.go:19-40): Fable 5 orchestrates; review/chat/fallback worker seats are Sonnet/Opus tier ONLY; Haiku is refused outright; effort floor is medium (flooredEffort clamps up with a logged warning); orchestrator/deliverable default effort high.
- RESILIENCE: keyless degradation everywhere (anthropic selection without ANTHROPIC_API_KEY degrades to the legacy worker; stub runner writes an honest 'not configured' artifact); refusal retried once on the fallback model with provenance stamping; truncated turns (max_tokens) and turn-cap exhaustion produce needs_attention artifacts rather than silent success; the ambient framework (agent_runner.go — a separate cron-style system, not an AgentRunner) adds exponential backoff (30s→10m), window-halving retries, dead-lettering after 4 failures with coverage tombstones, and per-(agent,room) run locks; HTTP errors are logged server-side and surfaced status-only so API bodies never reach the browser.
- NAMING TRAP: there are two unrelated 'runner' systems in this flat package. agent_runner_iface/providers/anthropic + agent_thread_runner/followup = the on-demand AgentRunner seam (agent threads, /goal). agent_runner.go = the ambient/scheduled agent framework (brain, digests, ledgers) on OpenAI. tool_registry.go contains NO LLM calls — it is the 12-tool packaging-goal taxonomy (prompts, rubrics, palette payload); the orchestrator's actual tool schemas come from app.kanbanTools() filtered by orchestratorToolAllowlist in agent_runner_anthropic.go.

## AUDIT: textAudit

**Summary:** Audited all non-runner LLM call sites in the flat-package Go server. The system is dual-provider with key-presence routing: when ANTHROPIC_API_KEY is set, conversational/distillation surfaces run claude-sonnet-5 through one shared text seam (anthropic_text.go -> createAnthropicMessagesResponse in agent_runner_anthropic.go, always-streaming SSE, prompt-cache breakpoints, no retries); when keyless-Anthropic, the same call sites fall back byte-for-byte to OpenAI gpt-5.5 via the Responses API (openai_responses.go). Ambient memory workers (brain, board, slop) are OpenAI-only; taste/house-style distillers are Anthropic-only (silently never start keyless). Voice is OpenAI Realtime (gpt-realtime-2 + gpt-4o-transcribe); imagery is gpt-image-2. A founder 'routing doctrine' is enforced in code: never Haiku (doctrineModelOrDefault refuses haiku ids from env dials) and an effort floor of medium (flooredEffort clamps low/minimal UP), so every Effort:"low" passed on the Anthropic path is silently medium on the wire. brain_intake.go, board_drafts.go, and artifact_render.go contain zero model calls.

**Architecture notes:**
- KEY ROUTING SEAM 1 — anthropic_text.go: createAnthropicTextResponse(HTTP) is the single text-in/text-out wrapper every non-runner Anthropic surface uses (chat Q&A, memory recall, follow-ups, attachments, narrative maintainer, taste analyst, house style). It centralizes: chatModel() resolution (BONFIRE_CHAT_MODEL -> default claude-sonnet-5, never-Haiku guard), the doctrine effort floor (flooredEffort silently clamps low/minimal UP to medium — so every Effort:"low" caller is medium on the wire), thinking:{type:"disabled"} (matching the no-reasoning gpt-5.5 profile it replaced; must never be set for Fable 5), and hard errors on stop_reason refusal and max_tokens (no retry).
- KEY ROUTING SEAM 2 — agent_runner_anthropic.go createAnthropicMessagesResponse(HTTP): the one Anthropic wire path for the whole binary (text helper, Scout router, goal engine). Always streams SSE and accumulates; x-api-key auth from ANTHROPIC_API_KEY; anthropic-version 2023-06-01; optional beta header only via BONFIRE_ANTHROPIC_BETA. buildAnthropicMessagesPayload spends the 4 prompt-cache breakpoints as: last tool, system prompt, newest 2 user turns (image blocks never marked). Effort marshals as output_config.effort only when non-empty. NO retries/backoff anywhere in-repo — a single http.Client.Do; errors are status-only (upstream body logged server-side, never surfaced to the browser).
- KEY ROUTING SEAM 3 — openai_responses.go createOpenAITextResponse: the OpenAI Responses-API twin (https://api.openai.com/v1/responses) used by the keyless-Anthropic fallbacks and the OpenAI-only ambient workers; meetingBrainModel() (OPENAI_BRAIN_MODEL -> gpt-5.5) is its central model dial. Also no retries.
- ROUTING DOCTRINE (founder directive 2026-07-07, agent_runner_anthropic.go:19-51): Fable 5 orchestrates (runner side); all worker seats are Sonnet/Opus tier ONLY; doctrineModelOrDefault refuses any env model id containing 'haiku' (logs + keeps default) but accepts any other arbitrary string (a typo'd id would only fail at request time with a 404); no Anthropic surface runs below effort=medium.
- PROVIDER SPLIT PATTERN: 'Sonnet fronts X whenever an Anthropic key is present; keyless keeps the gpt-5.5 path byte-for-byte' — implemented as if currentAnthropicAPIKey() != "" at each call site (memory_query x2, agent_thread_followup, narrative_maintainer). Scout router, taste analyst, house-style distiller, and attachment transcription are Anthropic-only (feature silently absent keyless). Brain, board, and slop workers plus Realtime voice/transcription and imagery are OpenAI-only.
- FILES WITH ZERO MODEL CALLS (audited as requested): brain_intake.go (deterministic hardcoded interview script; synthesis rides the ambient brain worker), board_drafts.go (draft accept/dismiss mechanics only), artifact_render.go (sandboxed HTML render security model only), domain_terms.go (regex corrections + vocabulary; only defines realtimeTranscriptionModel() consumed by kanban.go), kanban.go's own LLM surface is the Realtime session config (its board/memory tools call back into the seams above).
- MAX-TOKENS BUDGETS were re-baselined ~+30% on the Anthropic path for the Sonnet 5 tokenizer (chat 800, follow-ups 3000 vs OpenAI 500/700/2600). The chat path disables thinking specifically because Sonnet 5 defaults to adaptive thinking when the field is omitted and max_tokens caps thinking+text combined — but the Scout ROUTER omits DisableThinking, so it does run adaptive thinking inside its 700-token tool-call budget (only such non-runner site).
- CADENCE/COST SHAPE: highest-frequency LLM traffic is (1) Realtime voice+transcription during live meetings, (2) per-chat-message pairs (router turn + answer turn = up to 2 Sonnet calls per private message), (3) 2-5 min ambient brain/board ticks per active room. Distillers (taste hourly-gated, house-style ~monthly, slop 6-hourly) are cheap tails. All workers are lazily started at first room admission (no always-on spend) except slop (boot-started from main.go) and taste/house-style (registered alongside first ambient agent).

## RESEARCH: veniceResearch

**Summary:** Venice.ai exposes a genuinely OpenAI-compatible API (base https://api.venice.ai/api/v1, POST /chat/completions, Bearer auth) with streaming, tools/tool_choice/parallel_tool_calls, json_schema structured output, and reasoning_effort control. GLM 5.2 (Z.ai, released ~June 13-16 2026; ~744B-total/~40B-active MoE, open weights) is live on Venice as model id "zai-org-glm-5-2" — flagged BETA, FP8-quantized, 1M context, 131,072 max output, function-calling and reasoning enabled, no vision — priced $1.40/$4.40 per MTok (cached input $0.26), identical to Z.ai's first-party API pricing. Independent benchmarks place GLM-5.2 as the best open-weight coding/agentic model: it beats GPT-5.5 on SWE-bench Pro (62.1 vs 58.6), FrontierSWE (74.4 vs 72.6) and Terminal-Bench 2.1 (81.0), but trails Claude Opus 4.8 on the hardest tasks (SWE-bench Verified ~82% vs 88.6%; SWE-bench Pro leader Opus 4.8 ~69.2%). Zhipu shipped no official benchmark suite at launch, so all figures are third-party. Venice publishes no SLA, has modest per-model rate tiers, markets contract-enforced zero-retention plus TEE/E2EE modes (no SOC2/GDPR certs cited), and last acknowledged an outage May 21, 2026. Tool-call reliability for GLM models varies sharply by host (OpenRouter measured ~8% vs ~1% error rates across providers), which is the single biggest operational risk for production agent routing.

## DESIGN: integrationDesign

**Summary:** Concrete Venice/GLM-5.2 integration design for the Bonfire agent layer. The system is hand-rolled stdlib HTTP with two wire seams (Anthropic Messages SSE, OpenAI Responses) and a clean 3-method AgentRunner interface behind a closed name enum. Venice is OpenAI chat-completions-compatible (base https://api.venice.ai/api/v1, Bearer auth, model id zai-org-glm-5-2, $1.40/$4.40/MTok, BETA, no vision, 1M ctx), so integration is a third hand-rolled client file plus doctrine-compliant routing dials — NOT a repoint of the existing OpenAI path (that speaks Responses, not chat-completions, with a hardcoded URL). Build order: (0) per-call token+cost telemetry ledger FIRST (both providers currently discard usage; routing decisions are blind without it), (1) a veniceTextResponder swapped into the no-tools ambient text lanes behind per-seat env dials with automatic same-call fallback to the incumbent provider, (2) strict-JSON lanes after parse-failure telemetry clears, (3) a venice_glm AgentRunner + Fable-emitted model_choice for /goal subtasks (validated exactly like assignedRunner is today), (4) optionally a full tool loop with Anthropic tool_use <-> OpenAI tool_calls translation. Static per-seat config ships first; Fable-decides-at-runtime second; an intake classifier is rejected (adds a model call to save a model call, and the codebase doctrine already rejected silent routing once). Never migrate: the Fable orchestrator loop (server web tools + thinking round-trip + prompt-cache economics), the Opus review/gate and refusal-fallback seats (must accept replayed Anthropic thinking blocks), anything vision (GLM-5.2 is text-only), user-facing chat/router seats initially, and the Realtime voice/transcription/image/embedding surfaces (no Venice equivalent). This is a hedge-plus-70%-cut on the ambient text lanes and an execution-tier option for /goal — not a halving of total spend, which is dominated by the un-migrated Fable and Realtime seats.

### (a2) Tool-call translation layer (needed only for the Stage-3/4 runner, not for the text lanes): Anthropic tool_use <-> OpenAI tool_calls

A Venice runner wraps each kanbanTools() entry as {type:"function", function:{name, description, parameters}} and filters by the same orchestratorToolAllowlist (agent_runner_anthropic.go:1018). The translation that must be written is the MESSAGE-shape mapping inside a new venice loop: (1) assistant tool_use block {id, name, input:object} <-> OpenAI tool_calls[] {id, type:"function", function:{name, arguments:JSON-STRING}} — arguments is a serialized string on the OpenAI side and an object on the Anthropic side, and GLM hosts are known to emit malformed argument JSON (~8% on bad hosts per OpenRouter's Exacto data; Venice publishes no number and serves FP8), so the executor must json.Unmarshal-validate and reject-with-error-tool-result rather than crash; (2) tool results: Anthropic packs all tool_result blocks into ONE user turn; OpenAI wants one role:"tool" message per tool_call_id — the dispatch call into app.applyToolCallArgs and the 24K-char result cap are reused unchanged, only the envelope differs; (3) thinking: GLM streams reasoning_content separately — store it for debugging if desired but NEVER echo it back into the conversation and never attempt to translate it into Anthropic thinking blocks; conversely a Venice runner must never receive replayed Anthropic thinking blocks (this is also why the refusal-fallback seat can't be Venice); (4) server tools web_search/web_fetch DO NOT EXIST on Venice in Anthropic form — Venice has its own enable_web_search venice_parameter, but it's a different contract (no pause_turn); a Venice runner therefore ships WITHOUT research-mode server tools, which is fine because Stage 3 scopes it to drafting/synthesis subtasks; (5) vision: GLM-5.2 has no image input — the runner's job-acceptance check must reject any job whose context carries image blocks (the 12-image budget machinery marks them) and fall back to Fable. Critically: NO cross-provider mid-conversation handoff, ever — a tool loop that starts on Venice finishes or fails on Venice; fallback restarts the run (see d). Do not build any Anthropic->OpenAI history converter for partial transcripts; it is lossy (thinking blocks, cache_control, server-tool blocks) and the audit's pause_turn/_20260209 lesson shows how brittle verbatim-replay contracts are.

### (d) Failure/fallback semantics: same-call provider fallback with provenance, malformed-tool-call retry, circuit breaker; never cross-provider mid-loop
Venice has no SLA, tight rate tiers, a beta model, and returns 429 for capacity exhaustion — fallback is not optional. Mirror the ONE fallback pattern the codebase already trusts (refusal retry, agent_runner_anthropic.go:113: replay identical request on alternate model once, stamp provenance): (1) SINGLE-SHOT TEXT SEATS (Stages 1-2): wrap the seat's call as tryVenice-then-incumbent — on Venice transport error, 429, 5xx, timeout, or empty/unparseable output, immediately replay the SAME request on the seat's incumbent responder (gpt-5.5 Responses for ambient lanes) and stamp fallback_used=venice_failed into the ledger entry and, where a worker stamps model provenance into artifact metadata (narrative_maintainer does), record the model that actually wrote it. Because ambient workers already self-heal via cursor-not-advanced retries, even a double failure just retries next tick — no new recovery machinery. Do NOT add in-provider retry/backoff loops in v1 (the codebase has none anywhere; keep the posture consistent) — the cross-provider replay IS the retry. (2) JSON-STRICT SEATS (Stage 2): a Venice response that parses but fails the seat's JSON contract counts as a Venice failure -> same-call fallback to gpt-5.5, and increments a per-seat parse-failure counter that gates staying on Venice. (3) RUNNER/TOOL-LOOP (Stage 3+): within a Venice run, a malformed tool_call gets ONE reject-and-retry (error tool result: 'invalid arguments, re-emit'), a second consecutive malformation or any wire failure terminates the Venice run; the goal engine then relaunches the SUBTASK from scratch on Fable (assignedRunner rewritten, fresh conversation) — never translate a partial Venice transcript into Anthropic message history; the artifact provenance footer records both attempts. This reuses the existing needs_attention/relaunch plumbing rather than inventing resumption.
