# Memory Architecture Study: Bonfire Company Brain vs. Cloudflare Agent Memory

**Date:** 2026-07-10 · **Tree state:** axx/main `4c03874` + uncommitted coverage-stamp work (kanban-card-107) · **Their source:** blog.cloudflare.com/introducing-agent-memory/ (text extraction; the post's figures/diagrams did not survive fetch — visual specifics treated as unavailable, not absent)

**Mission frame (everything below is judged against this):** the company brain is the compounding substrate of the whole OS. It must (a) always recall what happened, (b) link storylines over time, (c) know who thinks what about what, and (d) compound — every meeting/chat/artifact makes the system smarter, never lossier.

---

## 1. Executive verdict

- **Keep our architecture; graft their disciplines.** Our consolidation stack (narrative tiers → event-sourced entity ledger → verbatim anchors → coverage-honesty stamps) holds meetings, storylines, and provenance in ways Cloudflare's atomic keyed-fact store structurally cannot. They hold no storyline construct at all; we should not trade shape for their plumbing. But they practice five write/read disciplines we concretely lack, and all five are small-to-medium grafts onto primitives already in our tree.
- **Our single worst property is deletes-on-silence at T2.** The cumulative meeting digest is a full model rebuild trusted by prompt alone to carry facts forward (meeting_digest.go:424,447); the predecessor is archived AND hidden from recall (memory.go:172-180). A fact the model drops on rebuild vanishes from every briefing surface forever — a direct violation of "never lossier." Their invariant — supersession only on key collision, silence preserves facts — is the fix, enforceable in pure Go.
- **Our single worst ops property is that the entire company memory lives on one droplet's disk with zero backup.** `data/` survives redeploys (deploy/digitalocean/README.md:178) and that is the whole durability story. One disk failure = permanent total loss — the maximal violation of "always recall." Cheapest catastrophic-risk fix on the board.
- **Their one genuine retrieval superiority — semantic search — is our #1 recall failure.** We have zero embeddings, zero FTS, zero vector store anywhere (verified; memory.go:2059-2131 is a lexical substring scan + a 15-group static synonym table). "The Korean TV thing" vs "Samsung TV Plus" returns nothing, and the same title-token weakness silently forks ledger records on rename (entity_ledger.go:495-499). Their write-time alias trick plus a small in-process vector lane closes it without any vector DB.
- **Who-thinks-what is failed by both systems; we fail it less badly but still fail it.** We capture far more per-person raw evidence than anything in their post (speaker-attributed transcripts, decision madeBy, ledger Owner, ratification). But we have no (person, topic) stance store, decision supersession is manual-HTTP-only (decision_ledger.go:257-293 has zero automatic callers), and positions argued in Scout chat compound nowhere (memory.go:2055-2057). Their keyed-supersession + version-chain primitive is the missing piece — and we already own every component needed to build it.

## 2. Cloudflare's architecture, faithfully

All claims below trace to the extraction (`cloudflare-agent-memory-extraction.md`, §refs). The post's figures did not survive the text fetch; where a diagram presumably carried detail (pipeline topology, channel weighting), we treat it as unavailable rather than absent.

### 2.1 Memory model (§1)
The unit is a **profile** — "an isolated memory store shared across sessions, agents, and users," addressed by name. Four memory types:
- **Facts** — "atomic, stable knowledge like 'the project uses GraphQL' or 'the user prefers dark mode'"
- **Events** — temporal occurrences (deployments, decisions)
- **Instructions** — "procedures, workflows, runbooks"
- **Tasks** — "what is being worked on right now and are ephemeral by design"

Facts and instructions are keyed with topic normalization; "when a new memory has the same key as an existing one, the old memory is superseded rather than deleted." Tasks are excluded from vector indexing but discoverable via full-text search.

### 2.2 Write pipeline (§2)
Two write paths: **bulk ingestion at compaction** (conversation history handed over when the harness shortens context) and **direct tool use** — the model calls `remember()` on demand. Deterministic IDs: "SHA-256 hash of session ID, role, and content, truncated to 128 bits" → idempotent re-ingestion; "everything is written to storage using INSERT OR IGNORE."

Two parallel extraction passes: a full pass chunking at "roughly 10K characters with two-message overlap" (up to four chunks concurrently), and a detail pass (9+ message conversations) with overlapping windows targeting "concrete values like names, prices, version numbers, and entity attributes." **Eight verification checks** before storage: "entity identity, object identity, location context, temporal accuracy, organizational context, completeness, relational context, and whether inferred facts are actually supported." Vectorization is async/background; the embedding text "prepends 3-5 search queries generated during classification" — the write pipeline works for the reader.

Model split: extraction/classification = Llama 4 Scout (17B MoE); synthesis = Nemotron 3 (120B MoE, 12B active). Their stated lesson: "a bigger, more powerful model isn't always better… The synthesizer is the only stage where throwing more parameters at the problem consistently helped."

### 2.3 Storage (§3)
Durable Objects with SQLite (raw messages, classified memories, FTS indexing, transactional writes) + a Vectorize index, **one DO instance + one Vectorize index per profile** → "strong isolation between tenants without any infrastructure overhead." R2 is future tense: "snapshots and exports will go to R2." Superseded memories keep "a version chain with a forward pointer from the old memory to the new memory."

### 2.4 Retrieval (§4)
**Five parallel channels:** full-text search (Porter stemming); exact fact-key lookup; raw message search (unclassified fragments); direct vector search; HyDE vector search. Merged with Reciprocal Rank Fusion — "each result receives a weighted score based on where it ranked within a given channel"; fact-key matches weigh highest; ties broken by recency. A synthesis stage feeds top candidates to the big model. Temporal queries: "date math handled deterministically via regex and arithmetic, not by the LLM." Design rationale: "no single retrieval method works best for all queries, so we run several methods in parallel and fuse the results."

### 2.5 Temporal / contradictions (§5)
Relative dates are resolved to absolutes **at extraction** ("yesterday" → "2026-04-14"). Message line indices stored for provenance. Recency is a ranking tiebreaker. Evolving facts are handled by keyed supersession + version chains; "no explicit contradiction detection beyond the verification checks."

### 2.6 Sharing, API, portability (§7-9)
Profiles can be shared by teams ("knowledge learned by one person's coding agent is available to everyone: coding conventions, architectural decisions, tribal knowledge"), or between agents (review bot ↔ coding agent). API: `ingest`, `remember`, `recall`, `list`, `forget`; REST for external agents; session-affinity headers for prompt-caching. Portability posture: "every memory is exportable."

### 2.7 Our text-described equivalents (so the comparison is apples-to-apples)
Several of their headline mechanisms already have equivalents — sometimes stronger — in our tree:
- **INSERT OR IGNORE ≈ our seen-id map** (memory.go:1008,1738; entity_ledger.go:200) — duplicate ids silently no-op.
- **Deterministic IDs ≈ our natural-id-per-seam scheme** — STT event ids (memory.go:645-651), `run-log-<threadId>` (agent_thread_runner.go:667), `brain-intake-<messageId>` (brain_intake.go:216). Derived tiers mint time-based ids (kanban.go:54-57) but carry idempotency via cursor-on-artifact + upsert supersession instead.
- **Keyed supersession + version chains ≈ upsertDigest** — every still-current digest for (kind,key) archived + `supersededBy` in one fsync'd critical section (memory.go:1585-1677); narrative slugs append-then-expire (narrative_maintainer.go:384-391); artifact body edits journal version lineage (memory.go:264-287,788-800).
- **Contradiction semantics: ours are strictly stronger where they exist.** They admit "no explicit contradiction detection"; our entity ledger CLOSEs validity windows, never deletes, and mints "reopens <id>" records on restatement (entity_ledger.go:898-923).
- **Their model-split economics ≈ our effort dials** — gpt-5.5 effort-low producers (openai_responses.go:18,57), Sonnet-5 fronting narrative + answers (narrative_maintainer.go:275-302; memory_query.go:1450-1489). Their empirical finding validates our allocation.
- **Session affinity for prompt caching:** solves a distributed problem we don't have (one process = perfect affinity), and we already stamp `cache_control` markers on the Anthropic wire (agent_runner_anthropic.go:223,348-486). No gap.

## 3. Our company brain today — the honest map

### 3.1 The substrate
One append-mostly JSONL (`data/meeting-memory.jsonl`, memory.go:578-584) loaded whole into RAM behind a single mutex (memory.go:315-334). 26 kinds (memory.go:19-135 + signals.go:33, deal_room.go:44, files.go:54) split into recall-eligible knowledge (transcript, brain, digests T2/T3/T4, decision, narrative, reflection, run_log, os_artifact, file…) and UI-state kinds structurally barred from all search/model context (`isUIStateMemoryKind` memory.go:2055-2057). O_APPEND writes, no fsync except digest supersessions (documented crash window memory.go:1079-1090); in-place edits are full-file rewrites. Sidecars: meetings directory `data/meetings.json` (**cap 200 newest**, meetings.go:40-42), content-addressed blob store (blobs.go:1-56).

### 3.2 The pipeline (write → consolidate → read)
STT firehose → filler-filtered, speaker-attributed transcript append with provider-event-id dedupe (memory.go:645-651; speaker_attribution.go:208-317) → cursor-gated ambient tiers on a shared chassis (agent_runner.go:851): **T1 brain** write-ups preserving who-said-what with transcript-reference audit sections (brain_worker.go:157-166) → **T2 cumulative per-meeting anchored-JSON digest** with server clamps, verbatim transcript anchors, and server-authored coverage stamps (`coverage` ∈ {full, partial_late_start, partial_gaps, unknown}, `sittingStartedAt/EndedAt`, `externalMayPredateCapture` — meeting_digest.go:239-309,552-575) → **T3 deterministic Go day fold**, zero model calls (meeting_digest.go:657-741) → **entity ledger**: event-sourced canonical records with validity windows, deterministic matching, one batched LLM adjudication per pass for the 0.4-0.8 Jaccard band only (entity_ledger.go:456-512,729-758) → **T4 company digest** (ledger projection + thin narrative, company_digest.go). Beside the tiers: keyed **narrative dossiers** (one active per slug, mandated Timeline/Key-people/Concerns sections, meetingIds unions across sittings — narrative_maintainer.go:150-164,349-361), **decision ledger** with madeBy/ratification/supersession chains (decision_ledger.go), **reflections** tracking consensus forming/diverging over 7-day windows (meeting_digest.go:946-1027), run logs, taste/house-style distillers.

Read side: a fixed-priority lane assembly inside a 60-entry budget (`contextEntriesForQuery` memory_query.go:2278-2434) — ledger-first current-state lane (deterministic, no-model fallback, memory_query.go:1778-1799), pinned digest lane exempt from the cap, lexical search, time-range snapshot, roster participant lane, scored artifact lane, last-8-brains, tail. Cross-meeting briefings compose deterministically from current digests with a map-reduce fallback (cross_meeting_briefing.go; recall_mapreduce.go), now coverage-stamped per meeting ("· captured HH:MM–HH:MM (partial — began late)", cross_meeting_briefing.go:325-342).

### 3.3 What compounds
- **Dossiers** accrete dated timeline prose + full meetingIds provenance across sittings (narrative_maintainer.go:349-361, kanban-card-107); the Active-storylines strip is pinned into every assistant answer (memory_query.go:1287-1298).
- **Ledger records** accrete anchors/meetingIds, ratchet importance, close-never-delete, reopen explicitly (entity_ledger.go:621-647,898-923); rebuilt digests re-consolidate as no-ops.
- **T2 digests** carry facts forward cumulatively per meeting; T3/T4 refold from CURRENT digests so superseded inputs self-heal (meeting_digest.go:657-741; company_digest.go:63-80).
- **Decisions** are pinned verbatim, with attribution and dates, into every Scout answer (memory_query.go:1270-1283).
- **Coverage stamps** make every reader state what was actually captured — a capture-honesty concept with no analog anywhere in their post.
- **Deterministic read lanes**: current-state answers render with zero model calls (ledgerStatusAnswer memory_query.go:1778-1799; foldDayDigest is pure Go) — their every recall is stochastic synthesis, by their own admission ("LLMs are stochastic, even with temperature set to zero").

### 3.4 What is lossy (the honest ledger, from map-ingestion §4)
- **Never captured:** filler/assent speech dies pre-store with no raw lane beneath the filter (memory.go:641,2264-2288); recording-toggle-off spans; **Scout chat thread content — private AND team-channel — is UI-state, never searched, never in context** (memory.go:2055-2057), the biggest structural hole for mission (c); private-thread attachments (files.go:57-62); grill/goal deliberation except via artifacts/signals/decisions.
- **Supersession-shaped loss:** T2 rebuild deletes-on-silence (meeting_digest.go:424 is instruction-only; prior digest hidden via memory.go:172-180); **dead-lettering** permanently skips a poison window after 4 failed passes with log-only evidence (agent_runner.go:652-686) — coverage stamps can't flag it because the raw transcripts exist; entity-ledger misses in-place decision status flips (entity_ledger.go:360-365).
- **Capped/evicted:** meetings.json 200-record cap degrades old meetings to coverage=unknown/untitled; ledger anchors/meetingIds cap 12 oldest-drop — a 15-meeting saga sheds its origin meetings (entity_ledger.go:568-584); company-digest projection keeps newest 3; narrative bodies 6000 chars with model-discretionary timeline compression; digest section caps drop fact #13 of a dense day.
- **Identity drift:** a storyline renamed mid-life (<0.4 Jaccard) silently mints a SECOND ledger record with no after-the-fact merge pass (entity_ledger.go:495-499); dossier slugs fork under synonyms beyond the 8-dossier context window ("samsung-tv" vs "samsung-tv-plus", map-storylines §5.2).
- **Trust-the-model fields:** T2 anchors are only trimmed to 80 chars, `at` to 40, `by`/`owner` length-trimmed only (meeting_digest.go:261-299) — a fabricated anchor silently empties the drill-down; a wrong `at` misfiles a fact onto the wrong day in every briefing (factDay meeting_digest.go:591-597). The roster-grounding discipline exists one file over (decision_ledger.go:583-584) and simply isn't applied at T2.
- **Hard deletes:** own room-chat message delete has no tombstone (kanban.go:3929-3951); quarantine expiry hard-deletes after 30d (stub survives).
- **Ops:** no offsite backup of `data/` exists anywhere in the repo (verified; deploy/digitalocean/README.md:178 is the entire durability story); O(N) search cloning the whole store under one mutex per query (memory.go:2079-2081); non-fsync'd appends.

### 4.1 Write pipeline
Their pipeline is atomic facts through a verification gate; ours is discourse through consolidation tiers. For a meeting OS, atomic keyed facts cannot represent "Tyler pushed back on pricing; AJ held" — hedged multi-party disagreement has no keyed-fact form, and their post shows no cross-fact narrative or storyline construct; synthesis happens only at read time. Our write side produces things theirs structurally cannot: brains that "preserve who said what, never invent" with audit sections (brain_worker.go:157-166), dossiers with mandated Timeline/Key-people/Concerns sections (narrative_maintainer.go:150-164), reflections on week-scale dynamics (meeting_digest.go:946-1027). Our provenance is deeper: verbatim transcript anchors resolving through `transcriptWindowAround` (memory.go:1931-2007) vs their "message line indices."

But they are cleaner on loss-safety, in four specific ways we should copy: (1) **silence preserves facts** — their supersession fires only on key collision, while our T2 rebuild can silently drop anything (the sharpest "never lossier" violation in our tree); (2) **verification is a pipeline stage, not prompt text** — and everything their eight checks do probabilistically, we can do deterministically, because we hold server truth (meeting span, transcript id set, roster) a generic memory product doesn't; (3) **write-time retrieval enrichment** — their classifier emits 3-5 search queries into the embedding text; our producers could emit aliases into searchable metadata at zero added model calls; (4) **a second, deliberate write path** — `remember()` — where we are ambient-only, leaving no seam for "remember: we never ship Q4 launches" or for anything argued in chat to reach recall. One caveat on their SHA(session,role,content) hashing: at our raw tier, speaker attribution IS the payload — two people saying the same words must stay two entries — so content hashing belongs only at our id-less fallback seam (memory.go:648-651).

### 4.2 Retrieval
Channel mapping: their FTS ≈ our lexical scan (theirs strictly better — Porter stemming vs substring matching where "art" hits "startup", memory.go:2094-2106); their fact-key lookup ≈ our ledger-first lane (ours richer: validity windows, owner, anchor drill-downs, and it doesn't just weigh highest — it LEADS the prompt as declared-authoritative, memory_query.go:2036-2037); their raw-message channel ≈ our full-body search scan; their vector + HyDE channels ≈ **nothing on our side**; their RRF ≈ our deterministic lane priority — a hard prior, which is defensible: canon SHOULD outrank raw hits; it's their "fact-key weighs highest" taken to its logical extreme. The real loss isn't fusion, it's the missing semantic channel.

Per query class: they win fuzzy-topic outright (High/Med-High vs our Low/Med) and edge us on arbitrary atomic facts (their detail pass targets "concrete values"; ours can be found by search yet truncated out of the prompt by the 8KB cap, memory_query.go:2287-2313). We win composed-temporal ("what did I miss") — coverage-honest, provenance-stamped briefings their post gives no evidence they can produce — and current-state/status, deterministically. Both fail who-thinks-what; we fail it less. Their "the synthesizer is the only stage where more parameters helped" mildly indicts our answer seam pinned at effort=low/800 tokens (memory_query.go:1477-1478; anthropic_text.go:25).

### 4.3 Temporal / storylines
Test question: "How did X evolve over the last month and where does it stand?" Their answer: RRF retrieval pulls top-k event atoms + current keyed facts; the big model composes the arc **on demand, every time** — completeness degrades as the store grows relative to fixed top-k, tasks are "ephemeral by design," and nothing notices omissions (no coverage concept anywhere in the post). Our answer: a precomputed dated dossier timeline + a replayable dated ledger event log (full post-state per transition, entity_ledger.go:133-138) + digest spans — precomputation wins on completeness, cost, and consistency. Their own rationale ("tighter ingestion and retrieval pipelines… provide a better foundation for… temporal logic, supersession") describes what we built further than they did.

Three real findings against us: (1) **"over the last month" literally fails our range parser** — `relativeQueryTimeRange` handles only yesterday/today/this-week/last-week/N-min-hrs (memory_query.go:2480-2533); everything else degrades to no-range context and our best month-scale evidence (the digest lane) never activates; (2) **they resolve relative dates to absolutes at write; we don't** — our T3 fold buckets by model-supplied, unvalidated `at` stamps (meeting_digest.go:591-597), misfiling facts onto wrong days downstream; (3) **our version chains are richer than theirs and recall-dead** — superseded digests/narratives are hidden entirely (memory.go:172-180), `ledger_event` history is UI-state-excluded (memory.go:2055-2057), and `ledgerState` folds last-event-wins, discarding intermediates (entity_ledger.go:237-257). We record the evolution trail more faithfully than they do, then throw it away at read time.

### 4.4 Who-thinks-what
Test question: "What does Tim think about X, and did he change his mind?" Their single-user story is clean: keyed fact collides → supersedes → version chain records the flip with absolute dates. But the post is silent on diarization, per-speaker facts, and how a shared team profile tracks WHO holds a position — whether keys can encode (person, topic) is unknown. Our capture is far richer: energy-heuristic speaker attribution with frozen windows and guest-prefix enforcement (speaker_attribution.go:208-317; memory.go:2169-2226), decision madeBy roster-grounded ("unknown names are blanked, never invented", decision_ledger.go:583-584), Owner with drift semantics, honest hedging ordered in every downstream prompt (meeting_digest.go:425-426).

But: no (person, topic) store exists (map-storylines §3); `markDecisionSuperseded` has **zero automatic callers** — a spoken reversal lands as a second ACTIVE decision, co-pinned with the one it contradicts, resolved only by the model noticing dates (decision_ledger.go:257-330; memory_query.go:1270-1283); the participant lane is locked to a hardcoded 7-name roster (participants.go:21-29); and positions argued to Scout in chat never compound. The deep irony: we already own every primitive their design requires — `ledgerTitleKey` topic normalization (entity_ledger.go:410-412), supersededBy chains, batched adjudication with a "supersedes" verdict, validity windows. They composed those around the (key → supersede → chain) loop; we composed ours around cross-meeting dedup and left positions diffuse. Making positions first-class is a recomposition, not new infrastructure.

### 4.5 Ops / architecture
Their per-profile DO+Vectorize silos are anti-mission for us: our T3 day fold, entity ledger, and T4 company digest are deliberately company-global folds ACROSS rooms — that cross-room consolidation IS the compounding. Their answer to team knowledge ("everyone shares one profile") is structurally what our shared store already is, with finer room scoping on top (roomId stamps memory.go:1025-1027; roomScoped agents agent_runner.go:93-118; privacy canary private_chat_brain_contract_test.go). The honest cost: our isolation holds by filter discipline + tests, not storage boundaries — acceptable single-tenant, no story for a second company.

Where they're sturdier: transactional SQLite + indexed FTS vs our O(N) clone-under-mutex scan and non-fsync'd appends; async background vectorization vs our nothing; R2 snapshot posture vs our zero backups. Their model-split economics we already match via effort dials (and our ledger/T3/T4 are more frugal than anything in their post — deterministic-first with one batched adjudication call per pass). Portability today we win outright: the whole brain is three greppable disk artifacts vs a profile behind a future-tense export API.

## 6. Adoption roadmap

Organizing principle: **adopt their disciplines, not their architecture.** Every item below builds in our codebase on primitives that already exist. Phases ordered by (mission impact × cheapness); Phase 0-1 items are independent and parallelizable.

### Phase 0 — stop the bleeding (this week; both small)

**0.1 Nightly offsite snapshot of `data/`** — *gap #1, effort: small.*
Cron on the droplet (or in-process ticker): tar `data/` (JSONL + meetings.json + blobs + users/sessions), encrypt, push to DO Spaces/S3, 30-day retention ring; restore procedure documented beside deploy/digitalocean/README.md. The JSONL's append-mostly shape makes snapshots consistent-enough — worst case loses a day's tail vs today losing everything. Why it compounds: everything else on this list is pointless if the substrate can vanish.

**0.2 T2 carry-forward guard (silence preserves facts, in Go)** — *gap #2, effort: small.*
After `parseMeetingDigest`, diff the prior current digest vs the rebuild using the existing `decisionDedupeKey` token pipeline (entity_ledger.go:410-412), per section: any prior non-terminal fact whose key is absent gets re-appended with `carriedForwardByServer=true` metadata (respecting section caps, importance-ranked), or at minimum counted in a `droppedFacts` stamp. Pure Go inside the producer around `clampMeetingDigestPayload` (meeting_digest.go:239-309). Statuses stay model-owned; **existence becomes server-owned** — matching how `upsertDigest` already treats supersession as server-enforced. Pin with a test feeding a rebuild that drops a live decision. Why it compounds: converts T2's cumulative contract from prompt-trust to structure; the digest lane grounds every briefing.

### Phase 1 — quick wins (small, pure Go or schema-only; 1-2 days each)

**1.1 Deterministic verification trio at T2 clamp time** — *gap #7, effort: small.*
Their eight checks, distilled to the three we can do exactly because we hold server truth. Extend `clampMeetingDigestPayload`: (a) **anchor-exists** — blank any anchor not in the store's transcript-id set for this meeting; (b) **temporal accuracy** — blank/clamp any fact `at` outside the server-known span (already computed, meeting_digest.go:401-413) so `factDay` falls back to the digest day instead of misfiling; (c) **attribution grounding** — pass `by`/`owner` through `normalizeTranscriptSpeaker` exactly as decision_ledger.go:584 does, preserving Guest prefixes. Zero model calls; mirrors the A5 "date math never by the LLM" doctrine and the coverage-stamp pattern (server truth overrides model claims).

**1.2 Temporal vocabulary broadening** — *gap #6, effort: small.*
Extend `relativeQueryTimeRange` + regex table (memory_query.go:24-26,2480-2533) with pure-Go parsing for: last/this month (calendar month in MEETING_TIME_ZONE), month names ("in June", year-disambiguated), "N days/weeks ago", weekday names, absolute dates (YYYY-MM-DD, "July 3"). Mirror in `briefingRangeFromArgs` (cross_meeting_briefing.go:490-524). Required rider: the digest lane cap of 24 is sized to today's ranges (comment memory_query.go:2220) — month ranges need a raised cap or day-digest-preferred selection for ranges >7 days. Table-driven tests per phrase. Why it compounds: converts silent wrong-era degradation into our best-in-class composed briefings — the machinery already exists downstream of the parse.

**1.3 Write-time alias enrichment (their embedded-search-queries trick, ported lexically)** — *gap #3 (half), effort: small-medium.*
Extend the T2 digest schema (meeting_digest.go:153-201) and narrative schema (narrative_maintainer.go:150-164) with `aliases: [...]` (3-5 alternate phrasings per topic/storyline — "Samsung TV Plus" → "the Korean TV deal", "CTV partnership", "TKO"), clamped and stored as entry metadata. `store.search` (memory.go:2059-2131) matches alias tokens at synonym weight (+2); **critically, union aliases into `ledgerTitleKey` matching** (entity_ledger.go:456-512) so a renamed entity matches an existing record's alias set before the <0.4-Jaccard band mints a duplicate. Zero new model calls (producers already run per pass), zero infra. Why it compounds: turns the static 15-group synonym table (domain_terms.go:69-85) into a self-extending, content-specific one; fixes recall drift AND ledger rename-forks in one move; doubles as embedding text if 2.4 ships.

**1.4 Dead-letter tombstones wired into coverage honesty** — *gap #9, effort: small.*
When the runner dead-letters a head input (agent_runner.go:668-686), append an ambient audit entry (pattern: slop_pass stubs, memory.go:54-59) recording agent/room/skipped-window span; teach `meetingCoverageLabel`/`meetingCoverageDetail` (memory.go:1442-1482) to check for overlapping dead-letter stubs and emit a `partial_synthesis` label so briefings say "captured but not synthesized" instead of silently omitting. Closes the one hole kanban-card-107's stamps structurally cannot see; also gives future manual re-feeds a durable target.

**1.5 Content-derived fallback IDs + tombstone the chat delete** — *gap #13, effort: small.*
Replace the `transcript-<nano>` fallback (memory.go:648-651) with `sha256(meetingId|speaker|normalizedText)[:16]` so redelivery without an STT event id no-ops via the existing seen-map — their INSERT OR IGNORE, biting only where we lack natural ids. Keep provider ids primary; hashing stays speaker-scoped (two people, same words = two entries). Separately, give own-chat delete a stub matching slop discipline (kanban.go:3929-3951).

**1.6 Word-boundary token matching + light stemming** — *gap #12 (partial), effort: small.*
In `store.search` (memory.go:2094-2106), match against pre-tokenized entry text (uniqueMemoryTokens exists) with a small suffix stripper (s/es/ing/ed); apply the same discipline to the classifier markers (memory_query.go:1930-1954) so "know" stops triggering the board short-circuit. Kills the "art"→"startup" class that wastes slots in the 60-entry budget.

### Phase 2 — structural (medium; the mission-(b)/(c) unlock)

**2.1 `note_for_the_record` — the deliberate-write path (their `remember()`)** — *gap #5, effort: medium.*
Add a tool to the assistant/orchestrator toolset (defs kanban.go:~2128ff; allowlist agent_runner_anthropic.go:1011) appending a recall-eligible entry via `appendEntryForMeeting` with id `note-<threadId>-<messageId>` (idempotent like run-log), author stamped from the session user — an **author-certain** lane, no heuristic hedge. Kind = decision with status=proposed for stances/decisions, or a new `note` kind. Available from chat threads and voice; the explicit tool call IS the consent surface, so the private-thread privacy contract (private_chat_brain_contract_test.go) stays intact. Why it compounds: the only fix for the chat hole that doesn't breach privacy — positions argued in a Scout thread finally have an escape hatch into the ledger lane.

**2.2 Keyed position records + automatic reversal adjudication** — *gap #4, effort: medium.*
Two coupled pieces, both recompositions of existing machinery:
- Add entity type `position` to the ledger (entity_ledger.go:79-84), mandatory Owner, key = Owner + `ledgerTitleKey(topic)`. Feed by extending the decision-extraction schema (decision_ledger.go:119) with stated positions/leans + holder (it already extracts directional leans as status=proposed). Route "what does <person> think about <topic>" through the ledger-first lane filtered by Owner (memory_query.go:1528-1772). Who-thinks-what becomes O(lookup).
- Run the ledger's batched adjudication pattern (entity_ledger.go:729-758, verdicts same/different/supersedes, degrade-to-ADD) in the decision extraction pass: new decision vs newest-50 actives in the 0.4-0.8 band; verdict=reverses → stamp `status=proposed-supersession` for one-click human ratification via the existing machinery (decision_ledger.go:380-412). Propose-first, per our own doctrine: a missed supersession is recoverable, a false one is not. Closes the two-co-active-contradictory-decisions hole.

**2.3 Evolution lane — make version chains readable at recall** — *gap #8, effort: small+medium.*
Small half: extend `formatLedgerRecordLine` (memory_query.go:1653-1698) and the pinned decisions lane to render chains inline — "current: <statement> (madeBy X, date) — previously: <superseded statement> (until <date>)"; optionally keep a `pastOwners` list at merge time (entity_ledger.go:621-626, one field). Medium half: an evolution-query router beside `isCurrentStateQuery` ("how did X evolve/change", "history of X", "has X changed since") resolving the subject via `searchLedgerRecords`, then rendering that record's dated transition history deterministically from the `ledger_event` log (the fold's inputs already carry full post-states + At + Reason, entity_ledger.go:237-257) as a synthetic prompt-only entry (pattern: memory_query.go:1700-1717), with a no-model fallback sibling to `ledgerStatusAnswer`. Answers the month-scale evolution question deterministically — something Cloudflare cannot do without a synthesis call. Zero new storage; the data is already on disk.

**2.4 In-process semantic lane, RRF-fused (their channels 4-5 + fusion, minus the vendor)** — *gap #3 (other half), effort: medium.*
Register an "embedding maintainer" via `startAmbientAgent` (agent_runner.go:183-194 — the taste-analyst pattern: silently absent keyless), consuming knowledge kinds after a cursor, embedding ONLY the consolidated corpus {brain, meeting_digest, narrative, decision, reflection, run_log, artifact title+excerpt} — thousands of rows, not the transcript firehose. Sidecar `data/embeddings.jsonl` (never touching the main JSONL or cursors); brute-force cosine over an in-RAM float32 matrix (milliseconds at our scale on 2 vCPU — **no vector DB**). One new lane in `contextEntriesForQuery` between search and snapshot (memory_query.go:2340-2352), weighted-RRF-fused with the lexical candidates + recency tiebreak, while ledger/digest lanes stay pinned above (our stronger form of their "fact-key weighs highest"). Lazily backfill on boot. Their design directly validates the pairing ("no single retrieval method works best for all queries"). Aliases from 1.3 become the embedding text. Optional follow-on once stable: a HyDE probe in the assistant seams only (25-45s budgets, memory_query.go:17-18), skipped for voice and for ledger-owned current-state queries.

**2.5 Absolute-date resolution at consolidation (their write-time normalization)** — *gaps #6/#7 rider, effort: small.*
One instruction line in `meetingDigestInstructions` + the brain prompt ordering resolution of spoken relative dates ("yesterday", "next Friday") to absolute YYYY-MM-DD using the window dates already provided — backstopped by 1.1's server-side `at` clamp, so a wrong resolution is caught deterministically. Same for decision statement/context fields (decision_ledger.go:119). Prevents decision text rotting into ambiguity as it ages through tiers.

### Phase 3 — deferred, trigger-conditional

- **As-of reads over supersession chains** (medium): a `get_meeting_detail`-family drill-down tool walking supersededBy/previousVersionId chains to answer "what did we believe about X two weeks ago" — bypassing `memoryEntryHiddenFromRecall` only inside the explicit tool, never ambient search. Lower priority: 2.3's ledger lane covers status-shaped evolution first.
- **RCU-style search snapshot** (small): replace clone-under-mutex (memory.go:2079-2081) with an atomic.Pointer to an immutable snapshot swapped on write. Buys the 10× headroom before any substrate change. Do when store size or latency evidence appears.
- **SQLite/FTS5 substrate migration** (large): in-process (Dockerfile already CGO_ENABLED=1). Deliberate later migration under 26 kinds and every seam — triggered by scale signals, not adopted now.
- **Small-model dial on extraction tiers** (small): the seam exists (`OPENAI_BRAIN_MODEL`, openai_responses.go:57-62). Their 17B-extract finding says it might work; our never-cheap-worker doctrine + the gpt-realtime prompt-gate deploy incident say gate it behind a before/after digest-fidelity eval. Pure API-cost play, zero mission upside — last.
- **Provenance-cap relief** (small): raise ledgerMeetingIDCap/anchor caps or spill to an overflow metadata field before the oldest-drop, so month-long arcs keep origin meetings. Cheap, but decide alongside 2.3 (the evolution lane changes what the caps cost).

### Do NOT adopt (with reasons)

| Their piece | Why not |
|---|---|
| **Per-profile hard silos** (one DO + Vectorize per profile, §3) | Anti-mission. Our T3/T4/ledger folds are deliberately company-global ACROSS rooms — that cross-room consolidation IS the compounding (mission b, d). Their own team answer is "share one profile," i.e., what our shared store already is. Our visibility-class + roomId-filter model with test pins is the right shape, proven at the guest/private boundary. Revisit only if Bonfire ever hosts a second company. |
| **Four-type atomic fact store as the primary substrate** (§1) | Cannot hold discourse: no storyline construct, no cross-fact narrative, no multi-party disagreement representation; arcs reassembled by top-k every query, degrading with store growth. Our tiers exist precisely because atoms can't hold a meeting. |
| **Tasks-as-ephemeral** (§1) | "What we're working on right now" deliberately decaying is a compounding loss against mission (b). Our board + ledger action_items with validity windows are the right model. |
| **Vectorize / Durable Objects / R2 as products** (§3) | Vendor plumbing. Adopt the patterns in-process: sidecar embeddings file (2.4), any object store for snapshots (0.1). At our scale a vector DB is pure overhead. |
| **Session-affinity headers for prompt caching** (§8) | Solves a distributed problem we don't have; one process is perfect affinity, and `cache_control` markers already ride the Anthropic wire (agent_runner_anthropic.go:223,348-486). |
| **SHA-256 content IDs everywhere** (§2) | At the raw transcript tier, speaker identity is the payload — content hashing would collapse two people saying the same thing. Adopt narrowly at id-less fallback seams only (1.5). |
| **Immediate SQLite migration** (§3) | Right destination, wrong trigger. The substrate is under 26 kinds and every seam; migrate on scale evidence, not admiration. Interim: RCU snapshot (Phase 3). |
| **RRF over everything, replacing lane priority** (§4) | Our fixed priority is a feature for a company brain: ledger canon and digest evidence SHOULD outrank raw hits deterministically. Fuse only within the raw-candidate band (2.4). |
